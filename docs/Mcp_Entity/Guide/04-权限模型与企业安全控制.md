# 权限模型与企业安全控制

> 日期：2026-07-17  
> 证据等级：OpenClaw 官方（B）+ eClaw 官方安全卖点（A）+ 企业策略 schema 推断（D）

## 1. 先分清四层边界

OpenClaw 官方明确把控制面拆开：

| 层 | 配置位置 | 决定什么 |
|---|---|---|
| Skill 可见性 | `agents.*.skills` / `skills.entries` | Agent **看不看得到**某份说明书 |
| Tool policy | `tools.*` / `agents.list[].tools.*` | Agent **能不能调用**某类工具 |
| Sandbox | `agents.*.sandbox.*` | 工具 **在哪里跑**（宿主机 / 容器） |
| Exec approvals / Elevated | `exec-approvals.json` / `tools.exec.*` / `tools.elevated.*` | 宿主机命令要不要审批、能否逃出沙箱 |

官方硬规则：

- **Skill allowlist 不是 shell 授权边界。**
- **Tool policy 是硬停止。** `/exec` 不能覆盖被 deny 的 `exec`。
- **deny 永远优先于 allow。**
- **allow 非空时，未列出的一律视为阻止。**
- 后一层只能继续收紧，不能把前一层拒绝的能力“授权回来”。

## 2. 工具可见性过滤链

模型只能看到通过全部过滤后的工具 schema：

```text
tool profile
  → provider tool profile
  → global allow/deny
  → provider allow/deny
  → agent allow/deny
  → agent provider allow/deny
  → sandbox tool policy（仅当处于沙箱）
  → subagent tool policy
  → plugin / MCP availability
```

常用 group 简写：

| Group | 包含 |
|---|---|
| `group:runtime` | `exec`, `process`, `code_execution` |
| `group:fs` | `read`, `write`, `edit`, `apply_patch` |
| `group:sessions` | sessions_* / subagents / session_status 等 |
| `group:memory` | `memory_search`, `memory_get` |
| `group:web` | `web_search`, `x_search`, `web_fetch` |
| `group:ui` | `browser`, `canvas` |
| `group:automation` | `heartbeat_respond`, `cron`, `gateway` |
| `group:messaging` | `message` |
| `group:nodes` | `nodes`, `computer` |
| `group:agents` | agents/goal/plan/skill_workshop 等 |
| `group:media` | image / music / video / tts 等 |
| `group:plugins` | 插件工具 + 经 `bundle-mcp` 暴露的 MCP |

重要提醒：

- 允许 `exec`、禁止 `write/edit`，**不等于**只读；因为 shell 仍可改文件。
- 若显式 allowlist 过滤后没有可调用工具，OpenClaw 会在提交模型前失败，而不是退化成纯文本 Agent。

## 3. 沙箱模型

```json5
{
  agents: {
    defaults: {
      sandbox: {
        mode: "non-main",   // off | non-main | all
        scope: "agent",     // agent | session | shared
        workspaceAccess: "rw" // none | ro | rw
      }
    }
  }
}
```

| 模式 | 含义 |
|---|---|
| `off` | 全部在宿主机跑 |
| `non-main` | 非主会话进沙箱；群聊/频道常踩坑 |
| `all` | 全部进沙箱 |

调试命令：

```bash
openclaw sandbox explain
openclaw sandbox explain --agent work --json
```

沙箱工具策略示例：

```json5
{
  tools: {
    sandbox: {
      tools: {
        allow: ["group:runtime", "group:fs", "bundle-mcp", "openmontage-doctor__*"],
        deny: ["browser", "gateway"]
      }
    }
  }
}
```

## 4. Exec Approvals（宿主机命令审批）

审批文件：

```text
$OPENCLAW_STATE_DIR/exec-approvals.json
# 默认
~/.openclaw/exec-approvals.json
```

核心字段：

```json
{
  "version": 1,
  "defaults": {
    "security": "deny",
    "ask": "on-miss",
    "askFallback": "deny",
    "autoAllowSkills": false
  },
  "agents": {
    "main": {
      "security": "allowlist",
      "ask": "on-miss",
      "askFallback": "deny",
      "autoAllowSkills": true,
      "allowlist": [
        {
          "id": "B0C8C0B3-2C2D-4F8A-9A3C-5A4B3C2D1E0F",
          "pattern": "~/Projects/**/bin/rg",
          "argPattern": "^safe\\.py$",
          "source": "allow-always"
        }
      ]
    }
  }
}
```

### 4.1 `tools.exec.mode`

| 值 | 行为 |
|---|---|
| `deny` | 阻止宿主机 exec |
| `allowlist` | 仅白名单命令直接跑 |
| `ask` | 白名单策略 + miss 时询问 |
| `auto` | 可自动评审确定性匹配，miss 再人工 |
| `full` | 无审批直接跑宿主机 exec |

遗留字段 `tools.exec.security` / `tools.exec.ask` 仍可用。

### 4.2 ask / askFallback

- `ask=off|on-miss|always`
- `askFallback=deny|allowlist|full`
- 无 UI 可提示时，默认按 `askFallback` 处理，通常应设 `deny`

### 4.3 allowlist 细节

- 可按二进制路径 glob 或裸命令名匹配；
- 裸命令名只匹配经 `PATH` 调用的命令；
- `argPattern` 用 ECMAScript 正则限制 argv；
- 链式命令（`&&`）要求每段都满足；
- `autoAllowSkills` 可自动放行 Skill 声明的 CLI；生产环境建议关闭或严格控制。

### 4.4 Elevated

- Elevated **只影响 `exec`**，不额外赠送其它工具；
- 不能覆盖 tool allow/deny；
- `/elevated full` 是破窗路径，仍受 host approvals 更严策略约束。

## 5. 多 Agent 覆盖

企业场景通常按 Agent 分权：

```json
{
  "agents": {
    "list": [
      {
        "id": "main",
        "sandbox": { "mode": "off" }
      },
      {
        "id": "family",
        "sandbox": { "mode": "all", "scope": "agent" },
        "tools": {
          "allow": ["read", "message"],
          "deny": ["exec", "write", "edit", "apply_patch", "process", "browser"]
        }
      }
    ]
  }
}
```

规则：

- Agent 级 sandbox/tool 覆盖全局；
- 每一层只能继续收紧；
- 空 allowlist 会让该 Agent 直接失败，而不是静默退化。

## 6. MCP 相关权限要点

1. MCP server 定义存在 ≠ Agent 可见。
2. 还需通过 tool profile / allow-deny / sandbox gate。
3. 单服务器可用 `toolFilter.include/exclude` 再收紧。
4. OAuth 未完成时，OpenClaw 只会省略该 server，不让整轮 Agent 失败。
5. `openclaw security audit` 会在启用 MCP Apps 等敏感桥时告警。
6. 非 owner 聊天参与者不能通过 `/mcp` 修改 owner 控制的 MCP 配置（官方发行说明已强调）。

## 7. eClaw 企业安全层（官方公开）

伊登官网把 eClaw 安全能力概括为三重防线：

### 防线一：双向数据防护

- 上行：识别 + 实时脱敏
- 下行：回流检测 + 内容审计

### 防线二：统一可观测

- Dashboard：沙箱状态、告警趋势
- 告警中心：异常行为 / 配置 / 资源风险分级
- 日志审计：统一存储、检索、导出

### 防线三：远程管控

- 远程删除高危 Skill/MCP
- 安全策略批量推送，统一沙箱准入规则
- 应急响应秒级阻断，自动隔离威胁沙箱

对集成方，这意味着：

- 本地配置可能被企业策略覆盖；
- 被标记为高危的 Skill/MCP 可能突然消失；
- 仅通过本机 JSON 无法完整描述最终生效权限；
- OpenMontage 必须把“企业策略允许”当作上线前置条件。

## 8. OpenMontage 建议权限基线

### P0 doctor

```json5
{
  tools: {
    profile: "messaging",
    allow: [
      "bundle-mcp",
      "openmontage-doctor__*"
    ],
    deny: [
      "group:runtime",
      "browser",
      "gateway"
    ]
  },
  agents: {
    defaults: {
      sandbox: {
        mode: "all",
        workspaceAccess: "none"
      },
      skills: ["openmontage-router", "openmontage-gates-intro"]
    }
  }
}
```

### P1 零 Key 出片

```json5
{
  tools: {
    allow: [
      "bundle-mcp",
      "openmontage-doctor__*",
      "openmontage-media__*",
      "read"
    ],
    deny: [
      "browser",
      "gateway"
    ],
    exec: {
      mode: "allowlist",
      ask: "on-miss"
    },
    sandbox: {
      tools: {
        allow: [
          "bundle-mcp",
          "openmontage-doctor__*",
          "openmontage-media__*",
          "read"
        ]
      }
    }
  }
}
```

并配合：

- 项目写路径只允许 `OPENMONTAGE_PROJECTS_DIR`；
- 付费 MCP 必须 `dry_run` + `confirm=true`；
- Skill 中禁止塞密钥；
- 关闭或严格限制 `autoAllowSkills`。

## 9. 仍需实机确认的权限问题

1. eClaw 默认 sandbox mode 是 `off`、`non-main` 还是 `all`？
2. 企业控制台是否生成独立策略文件，还是改写 `openclaw.json` / `exec-approvals.json`？
3. “远程删除高危 Skill/MCP”是删本地文件、改 allowlist，还是运行时拦截？
4. MCP tool 名称在 eClaw UI 中是否保留 `server__tool` 前缀？
5. Windows 桌面伴侣是否使用不同审批文件格式（OpenClaw 已提示某些 node host 有 native policy）？
