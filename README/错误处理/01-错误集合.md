## 之前遇到的重要问题总结

### 1. BGM 素材无声（-91dB ≈ "不存在"）

**现象**：从 Pixabay 下的两个 MP3（clouds.mp3, piano.mp3）编码到 AAC 后只有 2kbps，几乎静音。

**根因**：这些文件虽然显示 32kbps MP3、时长 56s/35s，但 `ffprobe loudnorm` 检测到 `input_i: -inf`（集成响度无限低），`volumedetect` 显示 `mean_volume: -91.0 dB`——人耳阈值以下。**Pixabay 返回了无效的占位音频文件**（可能是内容被清理/删除后的遗留）。

**解决**：用 Python 实时合成了一条 64 秒的 ambient + 钢琴背景音乐。

**教训**：从第三方免费音乐库下载音频后，**必须先用 `ffmpeg -i file.mp3 -af volumedetect -f null null` 验证实际音量**，否则可能白忙。

### 2. FFmpeg 字幕滤镜在 Windows 路径上崩溃

**现象**：`subtitles=D:\workfile\...\sub.srt` 报错 `Unable to parse "original_size" option value`。

**根因**：FFmpeg 的 `subtitles` 滤镜解析参数时，把 `D:` 的冒号当成了选项分隔符 `option:value`，认为 `D:` 是 `original_size=D`，于是 `:\workfile\...` 被当成不合法的分辨率值。

**解决**：把 SRT 文件复制到当前工作目录（无盘符路径），然后用相对路径 `sub.srt` 引用。

**教训**：Windows 上 `D:\` 这种盘符对 FFmpeg filter 是毒药。策略：

- 用相对路径绕过
- 或用 Python subprocess（list args 不走 shell 解析）
- 或用 `cmd.exe` 的 bat 文件

### 3. PowerShell 的参数传递问题

**现象**：同样的命令在 PowerShell 里运行和在 cmd / Python 里运行行为不同。具体有三种问题：

| 问题             | 表现                                                         | 原因                                                         |
| ---------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **`&` 管道截断** | `&&` 后面的命令被当成字符串参数                              | PowerShell 把 `&` 作为调用运算符，cmd 风格的 `&&` 停在 cmd.exe 层 |
| **逗号被吃**     | `force_style=Fontsize=24,FontName=Arial` 中 `,` 后面的参数丢失 | PowerShell 中逗号是数组分隔符，除非用引号包裹                |
| **引号嵌套**     | `-vf "subtitles=...:force_style='...'"` 多层引号频繁出错     | 每种 shell 对单引号/双引号的转义规则不同                     |

**解决**：**全程改用 Python `subprocess.run()` + list 参数**。list 参数直接传递给 `CreateProcess`，完全跳过 shell 解析，参数是什么就传给 FFmpeg 什么。

### 4. FFmpeg `amix` 混音输出的码率灾难

**现象**：63 秒的 WAV PCM（12MB，1536kbps）编码为 AAC 时只产出 18-30KB，码率 2-4kbps。

**根因**：AAC encoder 检测到输入信号几乎全静音（-91dB），自动降到最低码率。`Qavg: 65536` 是最大值，说明量化器认为不需要任何 bit。

**解决**：不在 FFmpeg filtergraph 里混音 + 编码一步完成，而是**分两步**：

1. 先输出 WAV PCM（无损中间格式）
2. 再用 `-b:a 128k` 单独编码 AAC
   这样每一步都独立可验证，方便排查。

**教训**：FFmpeg 的一站式 filtergraph 在特定情况下（极低信号量输入 + AAC encoder 的自动比特分配）会产生反直觉的"无声"输出。把音频处理流程**拆成可验证的独立步骤**是更可靠的方式。
